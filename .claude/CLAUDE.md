## 一、库的身份

**这是什么**：一个用于管理 ARMA3 Zeus 模式直播剧情的导演知识库。
导演（用户）通过 Obsidian 管理剧情、角色、伏笔、场次，并在直播中实时执行。

**核心工作流**：
```
规划剧本 → 直播执行 → 记录偏差 → 同步状态 → 复盘总结 → 续写下一场
```

**当前进度**（截至 2026-04-04）：
- 重制版第一季进行中，每季固定 **10集**
- ✅ 已完成：S1-1《换皮》、S1-2《投名状》、S1-3《清道夫》、S1-4《清理门户》
- 🔴 规划中：S1-5《开箱》
- S1 剩余配额：**5集**（S1-5 至 S1-10）

## 二、分层架构（L0–L6）

```
L0  导演作战台        每次开工第一眼，控制台 + 仪表盘 + 导航枢纽
L1  事实层           稳定不变的世界观事实（人物档案、世界规则、时间线）
L2  当前状态层        实时动态状态（人物位置/关系、地点控制、未决事项）
L3  方法论层         导演手册、规范文档、模板库、AI提示词
L4  内容生产层        伏笔管理（活跃/休眠/已回收三层体系）
L5  场次工作区        执行层（场次文件夹 + 直播后文件）
L6  差异学习层        复盘总结、成功/失败模式库
```

### 读写优先级规则
- **L1** = 只读参考（稳定事实）
- **L2** = 直播后同步更新（实时状态）
- **L4/活跃伏笔.md** = 续写AI默认读取的唯一伏笔源
- **L5** = 主要写作区域（场次文件）

## 六、Obsidian 插件使用规范

### ✅ Kanban 插件（obsidian-kanban）铁律

> 违反任何一条必然报错，已被错误验证过。详见 `.claude/kanban-lessons.md`

**Frontmatter**：只保留一行，其余什么都不加
```yaml
---
kanban-plugin: board
---
```

**Settings 块**：JSON 裸写在 `%% %%` 之间，**绝对不加反引号**
```
%% kanban:settings
{"kanban-plugin":"board","lane-width":300}
%%
```

**文件内容**：只能有 `## 列名` 和 `- [ ] 卡片`，**不能混入 Dataview 代码块**

**复杂配置**（metadata-keys 等）：用 Obsidian UI 面板配置，让插件自动生成，不要手写

### ✅ Dataview 插件

**不能**放在 Kanban 文件内，必须放在独立文件（配套的 `场次追踪表.md`）。

**常用查询模式**：
```dataview
TABLE 标题, 执行状态, 直播日期
FROM "L5_场次工作区"
WHERE 场次 AND 执行状态
SORT 场次 ASC
```

**Kanban + Dataview 分离模式**（已验证有效）：
```
场次子看板.md    ← 纯 Kanban 工作流
场次追踪表.md    ← 纯 Dataview 查询
```

## 七、操作安全规范

### 读写原则
- **读之后再写**：用 Edit 前必须先 Read，不盲写
- **不覆盖事实层**：L1 文件修改须谨慎，需要明确用户指令
- **偏差记录优先于剧本**：直播后，实际发生的事件 > 原剧本规划

### 路径规范
- 库内文件用相对路径，**不加前置斜杠**
- 正确：`L5_场次工作区/S1/S1-1/单场摘要.md`
- 错误：`/L5_场次工作区/...` 或绝对路径

### NPC 使用禁令
- **不能让已死亡/离开的 NPC 出场**（查 L2/人物当前状态.md）
- **不能凭记忆编造 NPC 名字**（以 L1/人物档案.md 为准）
- **不能让 [未知] 视角信息以 [已知] 方式透露**

### 地图坐标禁令
- **禁止自行编造坐标**（如 205-118、212-087 等）——AI不了解实际地图，编出来的坐标对应什么地方完全不可预测
- **只使用导演直播中确认的坐标**（如 205-114、220-210）
- **未知地点一律用描述代替**：「废弃仓库群」「警局」「海边驿站」，由导演直播中自行落点

### 视角标注规范
- `[已知]`：玩家角色（小蛋）已知的信息
- `[未知]`：玩家角色不知道但叙事层已设定的信息
- `[推演]`：基于已知信息的合理推断

## 八·附：直播剧本时间颗粒度铁律

> 本条规范来自实际写作中的错误修正，必须严格遵守。

**禁止使用的写法**：
- ❌「小蛋在营地待了一段时间」
- ❌「过了几天，情况发生了变化」
- ❌「经过一段时间的潜伏……」
- ❌「导演根据节奏控制时间长短」（时间跳跃版）

**必须使用的写法**：
- ✅ 每个场景是上一个场景的**直接后果**，紧接发生
- ✅ 用**事件节点链**替代时间块：A发生 → A导致B → B触发C
- ✅ 所有「过渡」必须是具体的行动（走路、对话、接到任务），而非时间流逝
- ✅ 导演只控制「推进几个节点」，不控制「过几天」

**原因**：这是 ARMA3 Zeus 直播，时间观是实时的。观众看到的是连续发生的事件，不是小说里可以跳过的段落。任何时间跳跃都会破坏直播的临场感和真实性。

## 八·附2：剧本只写唯一真实时间线

> 本条规范来自实际写作中的错误修正。

**禁止使用的写法**：
- ❌ 「可能是A，或者B，或者C」
- ❌ 「选择①…→结果；选择②…→结果」（选项树式写法）
- ❌ 「自行决策」「导演根据情况决定」（占位符式写法）
- ❌ 「任务内容：XXX（待定）」「地点：某处」

**必须使用的写法**：
- ✅ 做出一个具体决定，把它写进剧本，就是这个，不是别的
- ✅ 用真实的细节填充事件：具体的对话、具体的人、具体的地点、具体的感官细节
- ✅ 张力、幽默、紧张感来自细节，不来自「选项列表」

**原因**：剧本是导演的执行脚本，不是设计文档。「有多种可能」是策划阶段的思维，剧本阶段必须已经选好了那条路，并且把它写活。导演在直播中可以临场偏离，但剧本本身必须是一条清晰的线，不是一棵选项树。

## 九、伏笔推进规则

```
🔴 极紧急  → 本场必须有实质推进，不能只是"提及"
🟡 中期    → 可以埋下触发条件，不需要正面推进
🔵 长线    → 只能作为背景色，不主动推进
休眠       → 不出现，除非收到明确激活指令
```

伏笔移动需要同步：
- `L4_内容生产层/活跃伏笔.md`
- `L4_内容生产层/休眠伏笔.md`
- `L4_内容生产层/已回收伏笔归档.md`
- `L4_内容生产层/全局伏笔总表.md`（索引）
